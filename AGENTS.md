# Agent Notes

- Run Bazel with the same output base a developer will use. Do not add ad hoc `--output_base=...` flags for routine build, test, or run verification unless the user explicitly asks for that cache location. A custom output base creates a separate action cache, so successful verification there does not imply `bazel build/test/run ...` will be warm. Using a separate `--output_base=...` is OK when intentionally running multiple Bazel commands in parallel to avoid the Bazel server lock; call that out when reporting results.
- Inside `nix develop`, prefer the `bazel` wrapper over direct `bazelisk`; it delegates to Bazelisk. The Nix shell creates or updates the ignored workspace `user.bazelrc` with `build --config=nix`, so direct `bazelisk` also works after entering the shell.
- Prefer default Bazel compilation mode for routine development builds, tests, runs, and Codex verification. Only use `-c opt` when the user explicitly asks for release behavior, performance measurements, production image validation, or a bug that depends on optimized artifacts.
- Run Rust and CSS formatting checks through Bazel with `bazel run //:format.check` so the check uses Bazel-managed formatter tools instead of ambient shell tools. Use `bazel run //:format` to rewrite files in place.
- Run strict Rust lint checks by building real targets with the clippy aspect enabled, e.g. `bazel build --config=clippy //...`. For wasm targets, use `bazel query 'kind("rust_(library|binary|shared_library) rule", //crates/client/... + //crates/shared/...)' | xargs bazel build --config=wasm --config=clippy`. The clippy config denies warnings and the all, pedantic, and nursery lint groups.
- Keep controls consistent across games where interactions are analogous. Prefer shared helper code for board input semantics, drag/toggle behavior, timers, status displays, and other repeated game UI logic instead of copying slightly different implementations.
